
### 1. UDP는 연결 상태를 추적하기 어렵다

* UDP는 **connectionless 프로토콜**이다.
* 3-way handshake, 세션 상태, 종료 개념이 없다.
* 방화벽 입장에서

  * 정상 통신인지
  * 응답 트래픽인지
  * 위조된 패킷인지
    판단하기 어렵다.
* 따라서 **stateful firewall과 궁합이 매우 나쁘다**.

→ 보안 장비는 통제 불가능한 트래픽을 기본적으로 거부한다.

---

### 2. IP 스푸핑과 반사 공격에 취약하다

* UDP는 출발지 IP 검증을 하지 않는다.
* 공격자는 출발지 IP를 쉽게 위조할 수 있다.
* 이로 인해 다음 공격이 가능하다.

  * UDP Reflection / Amplification 공격
  * NTP, DNS, Memcached 기반 DDoS

→ 방화벽 입장에서는 UDP 자체가 DDoS 공격의 주요 수단이다.

---

### 3. 신뢰성·무결성 보장이 없다

* UDP는

  * 재전송 없음
  * 순서 보장 없음
  * 흐름 제어 없음
* 악성 트래픽과 정상 트래픽의 구분이 어렵다.
* 침입 탐지, 세션 분석이 힘들다.

→ 보안 분석 관점에서 비용 대비 효과가 낮다.

---

### 4. DNS만 예외로 허용하는 이유

* DNS는 UDP 기반이지만

  * 포트가 고정되어 있다 (53)
  * 트래픽 패턴이 명확하다
  * 없으면 인터넷 사용이 불가능하다
* 방화벽·보안 장비는

  * DNS Query/Response 구조를 인식하고
  * 크기, 빈도, 방향성으로 필터링이 가능하다.

→ 위험하지만 **필수 서비스이기 때문에 제한적으로 허용**한다.

---

### 5. UDP 위에서 동작하는 애플리케이션은 명시적 허용이 필요하다

* QUIC(HTTP/3)
* VoIP
* 스트리밍
* 게임 트래픽

이들은

* 포트
* 목적지
* 프로토콜 특성
  을 명확히 정의한 경우에만 허용한다.

---

### 핵심 요약

| 이유        | 설명                    |
| --------- | --------------------- |
| 상태 추적 불가  | 세션 개념이 없어 방화벽 통제가 어렵다 |
| 스푸핑 가능    | 출발지 위조로 DDoS 공격에 악용된다 |
| 보안 분석 어려움 | 정상/비정상 트래픽 구분이 힘들다    |
| 공격 빈도 높음  | 대규모 반사 공격의 주요 수단이다    |
| DNS는 필수   | 제한된 조건 하에 예외적으로 허용한다  |

---

### 한 문장 정리

DNS를 제외한 UDP 트래픽이 차단되는 이유는 **UDP가 연결·신뢰·추적 개념이 없어 보안 장비 입장에서 위험하고 통제 불가능하기 때문**이다.
